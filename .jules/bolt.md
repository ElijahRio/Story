## 2024-05-24 - Hoisting Regex and Eliminating Date Parsing Overhead
**Learning:** In highly iterated utility functions like `parseDateString` or calculations mapped across many objects (like `getAge` used during timeline rendering), defining regex literals inside the function `const parts = str.match(/(\d.../)` forces the JS engine to recreate the RegExp object on every call. Additionally, constructing Date objects from formatted strings (e.g., `new Date(`${y}-${m}-${d}`)`) introduces expensive string allocations and forces the browser to run heavy internal datetime parsing routines, which becomes a bottleneck in O(N) list renders. Also, parsing 'YYYY-MM-DD' as a standard date string results in a UTC midnight timestamp, but calling local time getter methods (e.g., `.getFullYear()`) introduces timezone shifting bugs.
**Action:** Always hoist static Regular Expressions into module-level constants (e.g., `const DATE_PATTERN = /.../`). When generating dates from integer parts, skip the string parser entirely by using the primitive `new Date(Date.UTC(y, m, d))` method, and ensure calculations on those dates use corresponding `.getUTC...` methods to preserve deterministic behavior and optimize performance.
