```javascript

  const textLinksCacheRef = useRef({ namesHash: '', cache: new Map() });

    // Generate a quick hash of all entity names to know when we MUST invalidate all matchers
```

Wait! Is `getDetectedLinks` dependency changing every time?
Yes! `getDetectedLinks` has `entityLinkDictionary` in its dependency array.
And `entityLinkDictionary` returns a NEW array on EVERY `entities` change!
So `getDetectedLinks` reference changes every time `entities` changes.
So `useMemo` will rerun EVERY time `entities` changes.
But inside the `useMemo`, `cachedEntry.text === allText` check will succeed for 99.9% of entities, meaning it skips `getDetectedLinks(allText, entity.id)` for them, completely bypassing the O(N^2) matching and making it O(N)!

Wait, if we put `textBasedLinks` `useMemo` at the root component level, it will run on EVERY keystroke!
Previously, the O(N^2) computation ONLY ran when `selectedId === 'network'`.
If the user never clicks the network graph, it never runs!
BUT when they DO click it, the UI freezes for 3 seconds.
And if we put it at the root component level in a `useMemo`, it runs on EVERY keystroke, so the user might experience lag when typing (even O(N) might take 10-20ms, which is fine, but still).
Actually, wait, if `getDetectedLinks` takes 30,000ms for 1000 entities, that's 30ms for 1 entity. So typing a character will take 30ms, which is perfectly smooth!
But generating the FULL graph on first click takes 30,000ms (30 seconds)!
By using the cache, the FIRST keystroke might take 30s?
No! When the app loads, `entities` changes once. The `useMemo` will calculate it once (taking 3s for 2000 entities, or whatever), delaying initial render.
Wait, we shouldn't delay the initial render if they are not viewing the graph!
So maybe we should NOT put it at the root component `useMemo`.
We can just put the caching logic INSIDE the `useEffect`!

This is brilliant!
If the user switches from a normal view to `network` view, the `useEffect` runs.
The first time it runs, it populates the cache.
If the user then edits a single entity and returns to the `network` view, it ONLY recalculates that single entity's text links!
This reduces the calculation from $O(N^2)$ to $O(N)$ when returning to the graph!
Wait! Is this correct?
If the user edits an entity, `entities` changes. Then they click `network`. The `useEffect` runs.
The `currentNamesHash` is checked. If they only edited the DESCRIPTION, `currentNamesHash` hasn't changed!
So the cache is STILL valid for all OTHER entities.
It only recalculates the single edited entity!
What if they edited the NAME?
Then `currentNamesHash` changes, the cache is cleared, and it does a full recalculation. But name changes are rare compared to description typing!
This is a MASSIVE performance win!
